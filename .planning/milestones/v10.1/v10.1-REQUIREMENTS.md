## Web Frontend: Painel do Vendedor e Chat

> **Branch ativa:** `feat/web-seller-chat`
> **Status:** Em planejamento
> **Iniciado em:** 2026-06-30

---

## Contexto e Motivação

Com a estrutura de fundação Web separada do Mobile na Milestone 10, o `web-frontend` agora roda isoladamente usando Next.js. O objetivo da Milestone 10.1 é equiparar a Web aos dois fluxos principais que faltam para o sistema ser end-to-end na web:
1. **Cadastro e Gestão de Veículos/Sucatas (Vendedor)**
2. **Chat em tempo real e Negociação**

Ambos já estão operantes na API em Java (em Produção) e no app Mobile. O foco aqui é exclusivamente a criação de interfaces ricas, performáticas e adaptadas para Desktop, prezando pelos melhores padrões de UI/UX (ex: Dashboard e Wizard/Stepper limpo) para o usuário web. A API de Produção será a fonte da verdade para o consumo local.

---

## Requisitos da Milestone 10.1

### R-10.1-01 — Configuração para Testes com a API em Nuvem
- **Critério:** O `web-frontend` localmente deve ser configurado via `.env` para consumir a API REST e as chaves do Supabase em Produção (`https://api-pecae.italohub.cloud/api/v1`).
- **Critério:** Os fluxos JWT de CORS devem estar alinhados para testar localmente contra a nuvem.

### R-10.1-02 — Área do Vendedor: Dashboard de Anúncios (Web)
- **Critério:** Rota `/vendedor/dashboard` restrita ao perfil Vendedor/Ambos.
- **Critério:** Deve exibir métricas resumidas (quantidade de anúncios ativos/inativos) e uma listagem (Data Table) robusta dos veículos do usuário logado, usando paginação ou infinite scroll.
- **Critério:** Ações na tabela para Editar (M10.2) e Inativar um anúncio.

### R-10.1-03 — Área do Vendedor: Assistente de Cadastro de Sucata (Stepper UI)
- **Critério:** Rota `/vendedor/anunciar` contendo um Wizard (Stepper UI) elegante e claro, semelhante a painéis premium do mercado automotivo.
- **Critério:** Steps: 1) Placa/Chassi; 2) Seleção FIPE (Marca, Modelo, Ano); 3) Peças disponíveis (Lista de seleção múltipla); 4) Upload de Fotos (Drag & drop zone com preview); 5) Preço e Condições.
- **Critério:** O estado deve ser retido durante os steps (react-hook-form + zod resolver).
- **Critério:** A submissão deve enviar o payload multiparte adequado (incluindo imagens) para a API já funcional de criação de anúncio, com indicação clara de que o anúncio foi para Moderação.

### R-10.1-04 — Interface de Chat e Negociação Web
- **Critério:** Rota `/chat` exibindo um layout de "Inbox" clássico (lista de conversas à esquerda em sidebar, histórico de mensagens à direita).
- **Critério:** Capacidade do Vendedor e do Comprador interagirem, consumindo os endpoints REST de chat e a camada de WebSocket/SSE para atualizações de tela em tempo real sem F5.
- **Critério:** Botão "Enviar Mensagem" na página de Detalhes do Veículo deve iniciar um novo canal de conversa ou redirecionar para um canal existente no chat, usando a API atual.

## Restrições e Decisões de Design

| Decisão | Escolha | Justificativa |
|---------|---------|---------------|
| **Formulário Web** | Stepper / Wizard | Maior conversão e menor carga cognitiva em desktop comparado a um form long-scroll |
| **Ambiente de Teste** | Frontend Local ↔ API Nuvem | Acelerar desenvolvimento sem precisar rodar todo o cluster Docker local pesadamente |
| **Moderação** | Mantida como está | Backoffice e Moderação Web avançada serão tratados em M10.2 |
| **Mecanismo de Busca** | Postgres FTS mantido | OpenSearch postergado pois não é prioridade primária de UI de Vendedor/Chat agora |
